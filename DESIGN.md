# AgentRoom 設計・要求整理

このドキュメントは、雑多な要求を「何を作るべきか」という形に整理したものです。実装方針を確定するものではなく、論点を明らかにして次の設計判断につなげるための土台です。

## 1. 背景・課題

- 現在、別々のセッションで動くAI（例: AI1, AI2）同士が会話する手段がない。
- Claude Codeにはバックグラウンドプロセスの出力を監視する「Monitor」機能や、イベント発生時に処理を起動する仕組み（hooks等）があるが、それを使って「相手からのメッセージが来るまで待機し、来たら自動で応答する」という会話ループはまだ組んでいない。
- 従来、AI同士・AIと人間の共同作業でのタスク管理はMarkdownファイルへの追記で行われることが多かったが、
  - 人間にとって読みにくい
  - 作業が進行中にAIがファイルを書き換えている最中に人間が横からタスクを追加するのが難しい（競合・タイミングの問題）
  という課題がある。

## 2. 実現したいこと（要求の整理）

### 2.1 AI間メッセージング
複数のAIエージェント（AI1, AI2, …）が同じサーバーを経由してメッセージを送受信し、会話できる。

### 2.2 イベント駆動の自動応答
AIエージェント側が、相手からの新着メッセージを検知したら自動的に反応（応答を生成して送信）する。人間が毎回「返信して」と指示しなくても会話が回るようにする。

### 2.3 ブラウザUIによる可視化・操作
サーバーが会話の状態をブラウザに送信し、人間がブラウザ上で
- 会話の内容を確認する
- 会話に介入する・操作する（詳細は「5. 未確定事項」参照）
ことができる。

### 2.4 タスク管理システム
AIが共同作業する際のタスクを、Markdownベースではなく構造化されたシステムとして管理する。
- 人間が読みやすい形でタスクの一覧・状態を確認できる
- 作業中でも人間がタスクを追加・編集できる
- Claude自身の詳しい思考過程を見たい場合は、元のClaude Codeのセッション（会話ログ）を見ればよい。AgentRoom側は「タスクの状態」と「AI間のやりとりの要点」を扱う。

### 2.5 （オプション）Claude CLI全文ミラーリング
可能であれば、Claude CodeのCLI上に表示されている文字列をすべて自動でこのシステムに転送し、ブラウザ側でもClaudeの発言をそのまま追えるようにする。

## 3. 全体アーキテクチャ（イメージ）

```
[AI1: Claude Codeセッション]  <--->  [Relay Server]  <--->  [AI2: Claude Codeセッション]
                                        |
                                        |--- 状態送信 --->  [ブラウザUI] <--- 操作 --- [人間]
                                        |
                                        |<--- 参照/更新 --->  [タスクストア]
```

- Relay Server: AI間のメッセージ中継、会話履歴の保存、タスクデータの保持、ブラウザへの状態配信を担う中心的なコンポーネント。
- AI側（AI1, AI2）: Claude Codeのセッションが、Relay Serverに対してメッセージを送受信するクライアントとして振る舞う。

## 4. コンポーネント別の要件と論点

### 4.1 Relay Server（メッセージ中継サーバー）
- 役割: エージェント間のメッセージ送受信・履歴保存・配信のハブ。
- 論点:
  - AI1↔AI2の1対1限定か、将来的に複数AI・複数ルームに拡張するか。
  - 通信方式（WebSocket / Server-Sent Events / 単純なポーリングAPI）はどれにするか。
  - ローカル環境だけで動かすのか、外部公開する可能性はあるか（認証の要否に直結する）。

### 4.2 イベント駆動の自動応答
- 目的: 新着メッセージが来たらAI側が自動で気づき、応答を生成して送り返す。
- Claude Code側で候補になる技術要素:
  - **Hooks**（`Notification` / `Stop` / `SessionStart` などのイベントで外部スクリプトを起動する仕組み）
  - **Monitorツール**（バックグラウンドプロセスの出力をストリーミングし、変化があれば通知を受け取る）
  - **ScheduleWakeup / Cron**（一定時間ごとに起動し直して新着有無を確認する、いわゆるポーリング的なアプローチ）
- 論点:
  - 常時待機（プロセスを立てたままイベントを待つ）にするか、定期実行（例: 数十秒おきに起こして確認）で妥協するか。
  - 応答までの遅延はどの程度まで許容できるか。
  - 複数のAIが同時に「待機」する場合、Relay Server側からどう通知するか（push型かpull型か）。

### 4.3 ブラウザUI
- 会話の状態（誰が何を話したか、いつ話したか）をリアルタイムに表示する。
- 人間からの操作候補（要具体化）:
  - 会話を一時停止/再開する
  - 人間が会話に一言メッセージを挟む
  - 会話の文脈（コンテキスト）を追加/修正する
  - タスクの追加・編集・状態変更
- 論点:
  - 「操作」の具体的な粒度（上記のうちどこまで必要か）。
  - ローカルのみで使うのか、他の人にも見せる可能性があるか（認証・アクセス制御に影響）。

### 4.4 タスク管理システム
- 課題: Markdownベースのタスク管理は、AIにとっては扱いやすいが人間には読みにくく、リアルタイムな共同編集が難しい。
- 要件（案）:
  - タスクをレコードとして構造化して保持する（id, タイトル, 状態, 担当者[AI or 人間], 依存関係, 更新履歴 など）。
  - ブラウザUIから人間がタスクをCRUDできる。
  - AI側もAPI経由でタスクを参照・更新できる。
- 論点:
  - Claude Code自体が内蔵しているタスク管理機能（TaskCreate/TaskUpdate等）と役割がどう違うか、連携するのか、それとは独立した仕組みとして作るのか。
  - タスクの粒度（AI1/AI2それぞれの作業単位のタスクか、会話全体を貫く共通タスクか）。

### 4.5 （オプション）Claude CLI全文ミラーリング
- 目的: Claude CodeのCLI上の表示内容をそのままRelay Server経由でブラウザに転送する。
- 技術候補: セッションのtranscriptファイルを監視して転送する、hooksの出力をキャプチャして送信する、など。
- 論点:
  - 元のClaude Codeのセッションを直接見れば済むケースと、あえてAgentRoom側にミラーリングする意義（ログの一元化、非開発者への共有、複数セッションの一覧性など）を整理する必要がある。
  - 必須要件か、あれば嬉しい程度のオプション機能か。

## 5. 未確定事項（実装前にすり合わせが必要な論点）

1. AIの数: AI1・AI2の2体固定を想定するか、将来的にN体（3体以上）に拡張する前提で設計するか。
2. 常時稼働の要否: Relay Serverやクライアント（AI側の待機プロセス）は常時起動させておくのか、人間が明示的に起動したときだけ動く想定か。
3. セキュリティ／公開範囲: ローカルPC内だけで完結させるのか、外部（別PC・スマホのブラウザ等）からアクセスする可能性があるか。
4. データの永続化方法: 会話履歴・タスクをどこに保存するか（例: SQLite、ローカルファイル、他のDB）。
5. 技術スタックの希望: 言語・フレームワークについて希望はあるか（例: Node.js/TypeScript, Python など）。
6. UIでの「操作」の具体的な内容: 4.3で挙げた操作候補のうち、どこまでを最初のバージョンに含めるか。
7. CLI全文ミラーリング（2.5）は必須要件か、後回しにしてよいオプションか。

## 6. 段階的な実装案（たたき台）

- Phase 0: 本ドキュメントでの要求整理・論点の確定（現在ここ）
- Phase 1: 最小のRelay Server + AI1/AI2間でのメッセージ送受信（自動応答は無しでもよい）
- Phase 2: イベント駆動 or ポーリングによる自動応答ループの実装
- Phase 3: ブラウザUIでの会話状態の可視化・基本的な操作
- Phase 4: タスク管理システムの統合
- Phase 5: （オプション）Claude CLI全文ミラーリング

## 7. 参考: 自動応答の起点として使えそうなClaude Codeの機能

- **Hooks**: `SessionStart` / `SessionEnd` / `Notification` / `Stop` / `PreToolUse` / `PostToolUse` などのタイミングで外部コマンドを実行できる仕組み。
- **Monitorツール**: 起動済みのバックグラウンドプロセスの標準出力をストリーミングで受け取り、変化があった時点で反応できる。
- **ScheduleWakeup / Cron**: 一定時間後、または定期的にセッションを再起動して処理を継続する仕組み（イベント即時性より定期確認に向く）。

どれを採用するか（または組み合わせるか）は「4.2 イベント駆動の自動応答」の論点に対応する。

## 8. 次のステップ

「5. 未確定事項」について回答を得たうえで、Phase 1の具体的なアーキテクチャ（通信方式・技術スタック・ディレクトリ構成）を確定させる。
